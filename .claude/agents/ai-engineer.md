## Step 3：AI 系统设计

你必须输出：

[artifact:AI架构设计]

必须包含：

- 模型层设计
- Prompt 层设计
- Context 管理设计
- RAG / 检索链路设计
- Tool 调用设计
- Memory 设计
- 安全与权限设计
- 监控与评估设计
- 降级与回退设计

需要明确：

### 1. 模型策略
- 使用什么模型
- 为什么选择该模型
- 哪些场景用主模型
- 哪些场景用轻量模型
- 是否需要多模型路由

### 2. Prompt 策略
- 系统提示词如何组织
- 角色提示词如何组织
- 任务提示词如何组织
- 输出格式如何约束
- 如何降低幻觉和跑偏

### 3. Context 策略
- 上下文从哪里来
- 保留多少轮会话
- 如何做摘要
- 如何控制 token 成本
- 如何防止上下文污染

### 4. RAG 策略
- 检索源是什么
- 向量库 / 索引如何设计
- 召回与重排是否需要
- 检索结果如何注入 Prompt
- 无结果或低相关结果如何处理

### 5. Tool Use 策略
- 需要哪些工具
- 触发条件是什么
- 工具失败如何回退
- 工具结果如何校验
- 是否允许多工具串联

### 6. 容错策略
- 模型超时怎么办
- 模型失败怎么办
- 工具失败怎么办
- 检索为空怎么办
- 输出不合规怎么办

### 7. 评估策略
- 如何验证回答质量
- 如何验证稳定性
- 如何做离线评估
- 如何做线上监控
- 如何做 prompt 迭代

# 五、工程与架构规范

默认技术方向：

- Node.js
- TypeScript
- Python（如有必要）
- OpenAI API / Anthropic API / 兼容网关
- LangChain / LlamaIndex（仅在确有必要时）
- Redis
- Vector DB
- Postgres / MySQL
- JSON Schema / Zod

工程规范：

- 严禁把 Prompt、业务逻辑、工具调用、数据处理混写在一个文件中
- 严禁把实验性 Prompt 直接作为生产 Prompt
- 严禁没有输出约束就直接依赖模型自由生成
- 严禁没有日志、监控和错误处理的 AI 接入
- 必须支持配置化模型切换
- 必须支持 Prompt 版本管理
- 必须支持错误追踪与响应日志
- 必须支持限流、超时、重试和熔断

# 六、AI 产品专项规范

你必须从 AI 产品视角关注以下问题：

## 1. 可解释性
- 用户是否知道 AI 为什么这么回答
- 是否需要引用依据
- 是否需要展示来源或检索片段

## 2. 稳定性
- 同类输入是否输出一致
- 是否可控制输出格式
- 是否可避免随机跑偏

## 3. 成本
- 单次调用成本是否可控
- 是否做了上下文裁剪
- 是否有轻量模型承担低价值任务

## 4. 安全
- 是否存在 Prompt Injection 风险
- 是否存在敏感信息泄露
- 是否存在越权检索与越权调用

## 5. 用户体验
- 是否有思考中提示
- 是否支持流式输出
- 是否支持中断与继续
- 是否在失败时给出明确反馈

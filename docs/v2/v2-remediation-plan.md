## 一、背景与问题定义

### 1.1 Why — 为什么需要修复

V2 版本完成部署后，对 7 个功能模块（R-01 ~ R-03、R-05 ~ R-08）进行全面审计，发现实现质量与原始设计稿存在系统性偏差：

- **前端 UI 偏离率**：7 个模块平均还原度约 75%，部分模块（R-02、R-03）仅完成约 60%~65%
- **后端 API 缺口**：存在 9 个 P0 缺失接口（核心业务流无法运转）和 4 个 P1 规范不合规问题
- **核心业务流断裂**：R-08 销售订单的状态流转（确认、发货、完成、关闭、触发建工单）全部缺失，订单流程无法闭环

### 1.2 What — 修复目标

1. **前端**: 100% 还原 `docs/v2/ui/*.html` 设计稿中的所有 UI 组件、交互状态、动画效果
2. **后端**: 补齐全部 9 个 P0 缺失接口 + 修正 4 个 P1 规范问题
3. **验收基准**: 每个功能模块对照 HTML 设计稿和交互说明文档逐项核查，达到 100% 通过率

### 1.3 How — 执行策略

按业务依赖关系和缺口严重程度分 3 个批次交付，优先修复阻断核心业务流的缺口。

## 三、修复批次计划

### 批次划分原则

- **Batch 1**（R-07 + R-08）：后端缺口最多，R-08 为核心业务流，R-07 是 R-08 的前置依赖（客户字段）
- **Batch 2**（R-01 + R-05 + R-06）：生产侧核心数据，R-06 依赖 R-05 的 taskId 接口
- **Batch 3**（R-02 + R-03）：采购侧增强功能，不存在跨批次依赖，可并行交付

## 七、非功能修复要求

### 7.1 全局 API 规范合规

所有 P1 规范问题（PUT→PATCH）须同时：
1. 后端修改路由方法注册
2. 前端修改对应请求方法
3. API 文档更新
4. 旧路由保留 30 天兼容期，返回 301 重定向，并在响应 Header 中添加 `Deprecation: true`

### 7.2 错误处理与容错

- 所有新增接口均须实现超时处理（前端 10s 超时，显示 Toast 警告+重试按钮）
- API 500 错误统一 Toast 提示：「服务器异常，请稍后重试」
- 网络断开时组件进入 offline 状态，恢复后自动 retry（最多 3 次，间隔 2s/4s/8s 指数退避）

### 7.3 响应式兼容

- 最小支持宽度：1280px（Web 端）
- 所有新增表格须支持横向滚动（overflow-x: auto）
- 新增 Drawer 组件须支持 Esc 键关闭和点击遮罩关闭

### 7.4 无障碍要求（WCAG 2.1 AA）

- 新增 Modal 和 Drawer 组件须实现焦点陷阱（focus trap）
- 所有状态变化须同时通过颜色 + 文字/图标区分（不能仅靠颜色）
- 闪烁动画须遵循 `prefers-reduced-motion` 媒体查询降级

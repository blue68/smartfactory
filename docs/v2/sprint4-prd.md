## 一、背景与目标

### 1.1 Why — 为什么做 Sprint 4

Sprint 1-3 已完成智造管家 V2 的数据基础与全链路贯通：

- Sprint 1：主数据补全（类目、价格、工价、Web任务管理）
- Sprint 2：BOM 通用化+版本化、客户管理、销售订单增强（插单）
- Sprint 3：采购→质检→入库→库存（R-09/R-11）+ 销售→生产→报工→交付（R-10）全链路打通

系统现已具备完整、可信的生产与采购数据。Sprint 4 是 V2 最后一个 Sprint，核心目标是在这份数据基础之上，实现 R-12 智能调度——将"人工每天查库存、对 BOM、判断要不要采购、决定排哪个订单先做"的脑力劳动，转化为系统主动给出有依据的调度建议，让工厂老板从日常运营决策中解放出来。

与此同时，Sprint 3 验收遗留了 4 项 P1/P2 技术债务（ACC-002/003/005/008），须在 Sprint 4 内并行修复。

### 1.2 What — 做什么

**核心交付（R-12 智能调度，规则引擎版本）**：

| 子能力 | 说明 |
|---|---|
| R-12-A 智能采购建议 | 在现有采购建议基础上，叠加安全库存补货规则 + 库存周转分析 + 资金占用评估，自动计算建议采购时机和建议数量，并展示计算依据 |
| R-12-B 智能排产建议 | 基于待排产工单的交期紧迫度、产能利用率和工序工时，给出工单优先级排序建议和工人产能分配建议，供车间主管参考 |
| R-12-C 调度看板 | 新增智能调度看板页面，整合采购建议和排产建议，展示规则引擎的计算步骤（"思考过程"可视化），让用户理解建议从何而来 |
| R-12-D 建议确认闭环 | 所有调度建议须经人工确认后才能转化为执行动作（采购建议→转PO，排产建议→调整工单优先级），禁止全自动执行 |

**并行修复（Sprint 3 遗留技术债务）**：

| 编号 | 内容 | 优先级 |
|---|---|---|
| ACC-002 | 质检单编号前缀统一（IQC vs QC，与业务方确认后执行） | P1 |
| ACC-003 | 引入消息队列（BullMQ/Redis Streams），彻底解决 EventBus 进程内同步阻塞问题 | P1 |
| ACC-005 | 退货单状态机命名统一（实现代码与 US 文档对齐） | P2 |
| ACC-008 | EventBus 异步 handler 异常增强告警（引入消息队列前的临时加固） | P2 |

### 1.3 How — 核心实现原则

1. **规则引擎优先，AI 模型延后**：MVP 阶段 R-12 完全基于规则引擎（确定性计算），不接入真实 LLM API。AI 模型版本待系统积累 4-8 周真实数据后再规划上线。

2. **建议必须展示依据**：任何调度建议必须在 UI 上展示完整的计算步骤（"AI 思考过程"实际是规则引擎的可视化分步说明）。禁止只给结论不给理由。

3. **人工确认是底线**：调度建议到执行动作之间必须有人工确认节点。禁止任何形式的自动下单、自动排产。

4. **建议置信度与数据基础透明**：每条建议需标注"基于近 X 天数据计算"和数据完整度评分，让用户理解建议可信度。

5. **ACC-003 消息队列引入是 Sprint 4 架构基础**：R-12 的异步调度计算将直接使用引入的消息队列，两项工作合并为同一架构升级。

## 四、非功能需求

| 类别 | 要求 |
|---|---|
| 性能 | 调度建议计算（采购建议 + 排产建议全量计算）完成时间 < 10 秒（基于 50 个 pending 工单、200 个 SKU 的典型数据规模）；单条建议详情计算步骤加载 < 1 秒 |
| 异步性 | 引入 BullMQ 后，入库接口响应时间 < 200ms（不含后台重评计算）；调度建议计算为后台队列任务，前端通过轮询或 SSE 获取计算完成状态 |
| 数据透明 | 每条调度建议必须展示：数据来源（基于哪些工单/SKU/库存数据）、计算时间戳、数据覆盖天数 |
| 可追溯性 | 所有人工确认操作（建议审批、转PO确认、排产调整确认）必须写入 audit_logs 表，含操作人、时间、原始建议快照 |
| 安全 | 调度建议页面按角色隔离：admin 可查看采购建议+排产建议；supervisor 仅可查看排产建议；purchaser 仅可查看采购建议；调度建议 API 全部需要 requireRoles 校验（修复 ACC-004 遗留问题） |
| 向前兼容 | 消息队列引入（BullMQ）须向前兼容现有 EventBus 接口，EventBus.publish() 方法签名不变，内部路由到 BullMQ；现有事件订阅者代码无需修改 |
| 数据冷启动处理 | 当系统数据积累不足（连续经营天数 < 14 天）时，调度建议页面展示"数据积累提示"，标注当前建议仅基于有限数据，置信度较低 |
| 可用性 | 消息队列 Redis 节点不可用时，系统降级为同步处理（进程内 EventEmitter 回退），不导致入库流程中断 |

## 五、技术约束

| 约束项 | 说明 |
|---|---|
| 技术栈 | 不引入新的后端语言/框架；消息队列使用 BullMQ（基于 Redis，现有系统已有 Redis）；前端不引入新的 UI 库 |
| AI 模型 | MVP 阶段禁止接入真实 LLM API（OpenAI/Anthropic）；所有"AI 建议"均为规则引擎确定性计算；前端展示"AI 思考过程"实为规则引擎步骤可视化，文案需诚实标注"基于规则计算" |
| 调度建议不可自动执行 | POST /purchase-orders（创建采购订单）接口只能由人工触发，禁止调度服务直接调用；排产服务同理 |
| 数据库兼容 | Sprint 4 新增表（schedule_suggestions、schedule_suggestion_items、suggestion_audit_logs）须包含 tenant_id 字段；不得修改 Sprint 1-3 已有表的主键和外键约束 |
| BullMQ 队列命名规范 | 所有队列名称使用 `erp.{domain}.{action}` 格式，如 `erp.inventory.shortage-recheck`、`erp.schedule.suggestion-calculate` |

| 前端状态设计 | 调度建议生成涉及异步等待，前端必须实现：计算中（loading 态）、计算完成（结果展示）、计算失败（错误恢复态）、无数据（空态）四种状态 |

## 七、User Story 完整列表

---

### US-S4-001 智能采购建议自动计算

**Story ID**：US-S4-001
**优先级**：P0
**前置依赖**：Sprint 3 R-11 缺料检测（MrpService）已就绪；库存数据、采购价格、供应商数据完整

```
As a 工厂老板 / 采购员
I want 系统自动综合计算每个 SKU 的最优建议采购数量和建议下单时机
So that 不需要人工查库存、算 BOM 用量、判断要不要补货，系统直接告诉我"需要采购什么、多少、为什么"
```

#### 验收条件

**AC-S4-001-01 规则引擎计算触发时机**
- 以下任一触发时机执行全量采购建议重新计算：
  1. 用户在调度看板手动点击"重新计算"按钮
  2. 每日定时计算（默认每天 06:00 执行，可配置）
  3. 新工单创建或工单排产状态变更后（异步触发，不阻塞主事务）
- 计算任务通过 BullMQ 队列（`erp.schedule.suggestion-calculate`）异步执行

**AC-S4-001-02 采购建议计算规则（规则引擎四步法）**

建议数量计算公式（按以下顺序执行）：

```
步骤一：缺口计算
  缺口量 = MAX(0, 各工单需求总量 - 当前可用库存 - 在途数量)

步骤二：安全库存补充
  补充量 = MAX(0, 安全库存阈值 - (当前库存 + 在途数量 + 步骤一缺口补采量))

步骤三：资金占用评估
  本次采购占用资金 = (缺口量 + 补充量) × 最近采购单价
  若占用资金 > 月度采购预算阈值（可配置），建议数量降为仅补缺口（不补充安全库存），并标注"资金占用预警"

步骤四：供应商推荐
  推荐供应商 = 有效报价（未过期）中综合评分最高的供应商（评分 = 价格权重 0.6 + 交货及时率权重 0.4）
  若无有效报价，建议状态为"待确认供应商"
```

**AC-S4-001-03 计算结果字段**

每条采购建议在原有字段基础上新增：

| 字段 | 说明 |
|---|---|
| engine_version | 固定值 `rule_engine_v1`（区分后续 AI 版本）|
| step1_shortage_qty | 步骤一计算的缺口量 |
| step2_safety_qty | 步骤二补充安全库存量 |
| step3_capital_warning | 布尔值，是否触发资金占用预警 |
| step3_capital_amount | 预估采购资金额（元） |
| step4_supplier_score | 推荐供应商的综合评分（0-100） |
| data_basis_days | 用于计算的历史数据天数（如：基于近 30 天数据） |
| calculated_at | 建议计算时间戳 |

**AC-S4-001-04 计算结果入库**
- 计算完成后，结果写入 purchase_suggestions 表，source = `ai_schedule`（延用 Sprint 3 已扩展的枚举值）
- 若该 SKU 已有 pending 状态的 ai_schedule 建议，更新其数量和步骤数据，不重复创建

### US-S4-004 智能排产建议引擎

**Story ID**：US-S4-004
**优先级**：P0
**前置依赖**：Sprint 3 US-S3-007（工序任务分配与排产）已就绪；R-05 工序工时数据已配置

```
So that 排产决策不靠经验和感觉，有数据依据，减少因排产不合理导致的交期延误
```

#### 验收条件

**AC-S4-004-01 排产建议计算规则**

排产建议基于以下维度计算优先级得分（100 分制）：

```
步骤一：交期紧迫度得分（权重 50 分）
  剩余天数 = 工单 planned_end - 今日
  剩余工时需求 = 工单未完成工序总计划工时
  工期余裕 = 剩余天数 × 每日可排产工时 - 剩余工时需求
  工期余裕越小（越接近 0 或为负），交期紧迫度得分越高

步骤二：销售订单优先级得分（权重 30 分）
  关联销售订单 priority = urgent：30 分
  priority = normal：0 分

步骤三：物料就绪度得分（权重 20 分）
  所有物料缺口为 0（物料齐套）：20 分
  有物料缺口但在途：10 分
  有物料缺口且无在途：0 分（降低排产建议，等待采购）
```

最终优先级得分 = 步骤一得分 + 步骤二得分 + 步骤三得分

**AC-S4-004-02 工人产能匹配建议**
- 对优先级最高的前 N 个工单（N 可配置，默认 10），系统给出工人分配建议：
  - 列出适合执行该工序的工人（基于工序类型，若工序有 skill_type 配置则按技能匹配）
  - 展示每个工人本周已分配工时 vs 可用工时（标注过载 / 正常 / 空闲）
  - 推荐空闲或正常工人，不推荐过载工人（标注"过载"红标，但不禁止选择）

**AC-S4-004-03 计算结果存储**
- 排产建议计算结果写入 schedule_suggestions 表（新增表）：
  - suggestion_type = `production_schedule`
  - 排序后的工单列表（JSON 数组，含工单 ID、优先级得分、各步骤得分、建议工人 ID 列表）
  - calculated_at、engine_version = `rule_engine_v1`

**AC-S4-004-04 排产建议刷新**
- 以下事件触发排产建议后台重新计算（BullMQ 队列，不阻塞主事务）：
  - 新工单创建
  - 工单 planned_end 修改
  - 工单优先级修改
  - 工序任务完工（产能释放）
  - 新缺料检测完成（物料就绪度变化）

### US-S4-009 库存周转分析面板

**Story ID**：US-S4-009
**优先级**：P1
**前置依赖**：Sprint 3 库存流水（inventory_transactions）数据已积累；US-S4-007 调度看板已实现

```
As a 工厂老板
I want 查看各 SKU 的库存周转天数和资金占用情况

```

#### 验收条件

**AC-S4-009-01 库存周转天数计算**
- 库存周转天数 = 当前库存数量 / 近 30 天平均日消耗量
- 近 30 天平均日消耗量 = SUM(过去 30 天 PRODUCTION_IN 和 SALES_OUT 类型的出库数量) / 30
- 若近 30 天无出库记录，标注"近期无消耗"（不计算周转天数）

**AC-S4-009-02 滞销预警**

## 一、背景与目标

### 1.1 V1 交付现状

V1 已于 2026-03-11 完成验收，核心成果如下：

| 模块 | 状态 | 说明 |
|---|---|---|
| SKU主数据（含两级分类） | 已实现 | 分类为系统预置固定枚举，不可自定义 |
| BOM管理 | 已实现 | 单版本，半成品不支持跨BOM复用 |
| 供应商管理 | 部分实现 | 后端接口完整，前端缺独立页面；无导出、无绩效对比 |
| 采购价格管理 | 已实现 | 无批量导入 |
| 工序管理 | 部分实现 | 无极限工时、无工价配置 |
| 小程序端任务管理 | HTML原型完整，原生小程序代码待开发 | |
| 销售订单 | 已实现 | 缺插单功能 |
| 采购流程 | 部分实现（建议+审批+订单跟踪）| 无完整来料质检入库链路 |
| 全链路数据打通 | 未实现 | 销售→生产→报工→交付链路断开 |
| 智能调度 | 未实现 | |

### 1.2 V2 业务目标

| 目标层次 | 具体目标 |
|---|---|
| 数据基础补全 | 类目可配、价格批导、工价核算，支撑主数据自主维护 |
| 业务中台打通 | BOM通用件+版本机制，消除重复录入和计算漏洞 |
| 全链路贯通 | 销售→生产→报工→交付、采购→质检→入库两条主链路完整闭环 |
| 智能决策升级 | 在全链路数据基础上，实现智能调度（采购建议+排产建议） |
| Web端能力对齐 | 消除Web端与小程序端的功能差距，Web端支持完整任务管理 |

### 1.3 核心约束

- 技术栈不变：React + TypeScript / Express + MySQL / 微信小程序
- 现有数据库表结构须向前兼容，不得破坏 V1 数据
- 每个 Sprint 2-3 周，共 4 个 Sprint，总周期约 10-12 周
- 智能调度（需求12）依赖全链路数据贯通，不可提前

## 二、需求分析

### R-01 SKU主数据 — 类目自定义配置

**业务价值**：中等偏高。工厂在实际使用 V1 后，会因业务扩展产生新品类（如"包材→气泡袋"、"辅料→加固件"等），固定枚举会频繁产生"无法归类"的操作障碍，客户投诉概率高。
**技术复杂度**：低。sku_categories 表已存在，V1 仅是固定枚举未开放 CRUD；需增加类目管理 CRUD 接口 + 前端配置页面 + 删除保护（有 SKU 关联时不允许删除）。
**依赖关系**：无前置依赖，可独立交付。
**风险**：需防止类目删除导致历史 SKU 分类孤立；子类目的父类目不允许随意变更（影响分类树联动）。

### R-02 供应商管理 — 导出 + 绩效对比

**业务价值**：中等。导出满足老板向外汇报/备档需求；绩效对比帮助采购员做选择，降低采购决策门槛。
**技术复杂度**：低（导出）+ 中（绩效对比）。导出使用 ExcelJS 即可；绩效对比需要在已有绩效数据基础上新增多供应商横向比较视图（含雷达图或条形对比）。
**依赖关系**：V1 供应商绩效接口已有，但前端 SupplierPage.tsx 未独立实现（V1 遗留 ⚠️ 问题），R-02 需先补全该页面。
**风险**：绩效数据若稀疏（新系统上线初期订单少），对比图表意义有限，需设计空态提示。

### R-03 采购价格管理 — 批量导入

**业务价值**：高。工厂存在大量历史采购价格数据，手工逐条录入体验极差；批量导入是加速数据迁移、降低录入成本的关键功能。
**技术复杂度**：中。需设计 Excel 模板、文件解析、字段校验（供应商编码/SKU编码匹配）、冲突处理（相同SKU+供应商+日期的价格如何处理）、导入预览、导入结果反馈。
**依赖关系**：依赖 SKU 主数据和供应商主数据已完整（R-01 补全类目后更好，但非强依赖）。
**风险**：脏数据导入问题；需明确价格生效规则（覆盖还是追加历史版本）。

### R-04 半成品通用化 + BOM版本迭代

**业务价值**：极高。这是 V1 数据模型的核心缺陷补丁。床尾板等通用件被迫在每个 BOM 中重复录入，一旦通用件发生变更（如材料规格调整）需要逐个 BOM 手动修改，极易遗漏导致生产计算错误。BOM 版本化解决不同生产批次口径不一的问题。
**技术复杂度**：高。涉及：
  - 数据模型变更：bom_items.component_sku_id 引用关系支持多对多（半成品 SKU 被多个 BOM 引用）
  - BOM 版本化：bom_headers 增加 version、effective_date、status 字段，需确保"激活版本"唯一性
  - 展开计算逻辑：采购需求计算引擎需按激活版本展开，防止跨版本计算污染
  - 操作体验：新建 BOM 时支持引用已有半成品（搜索选择，非重复录入）
**依赖关系**：影响采购需求计算引擎（F-201）、生产排产计算，是 R-10（销售→生产链路）的数据基础。
**风险**：最高风险项。数据模型变更需向前兼容（V1 存量 BOM 数据自动升为 v1.0 版本），展开计算逻辑需全量回归测试，任何漏洞直接导致采购建议错误。

### R-05 工序管理增强（极限工时 + 工价计算）

**业务价值**：中高。工价核算是生产成本管理的基础，老板和车间主管都需要这个数据做绩效发薪和订单报价参考。极限工时可用于排产时识别工序瓶颈。
**技术复杂度**：中。需在工序表增加 max_hours（极限工时）和 unit_price（计件单价）字段；完工上报时按件数 * 单价汇总工资；需要汇总报表视图（按工人/按工序/按时间段）。
**依赖关系**：依赖 V1 工序配置基础（F-007）；完工上报数据（生产任务完工记录）。与 R-10 生产链路相关，但可独立先做配置部分。
**风险**：计件单价涉及薪资数据，权限控制需严格（仅老板和车间主管可见汇总，工人仅能看本人数据）。

### R-06 Web端任务管理

**业务价值**：中高。目前 Web 端和小程序端存在功能差距——当操作人员在电脑前工作时无法执行完工上报等操作，造成使用摩擦。
**技术复杂度**：中低。V1 后端接口已有任务完工上报、异常上报等接口；主要工作是前端页面补全（新增 TaskPage.tsx、ExceptionReportPage.tsx 等）。
**依赖关系**：依赖 V1 后端生产任务接口完整性；R-10（生产链路）完成后 Web 端任务管理功能会更完整，但核心功能可提前交付。
**风险**：Web 端和小程序端数据一致性（同一任务被两端同时操作的冲突处理）。

### R-07 销售客户管理

**业务价值**：中高。V1 sales_orders 表已有 customer_id 引用，但客户主数据（customers 表）在 V1 中属于基础实现，缺少联系人管理、客户分级等完整功能。
**技术复杂度**：低中。标准 CRUD 模块，含联系人子表管理；难度不高。
**依赖关系**：是 R-08（销售订单）和 R-10（销售→生产链路）的前置。
**风险**：低。注意客户数据与订单的级联关系（删除客户时有历史订单的处理策略）。

### R-08 销售订单（含紧急插单）

**业务价值**：高。紧急插单是销售场景的高频需求，V1 已有常规订单，但缺少优先级抢占和排产影响评估。老板最关心插单后"哪些订单会延期"。
**技术复杂度**：中高。普通订单 CRUD 相对简单；插单功能需要：
  - 产能冲突检测（插单时实时计算当前排产是否有剩余产能）
  - 影响评估报告（哪些现有订单可能延期，延期多少天）
  - 需要老板或车间主管二次确认才能提交插单
**依赖关系**：依赖 R-07（客户管理）；与 R-10（生产链路）强关联，插单后需触发排产重新计算。
**风险**：插单影响评估算法复杂度较高，V2 阶段可先以"标记优先级+人工处理"方式简化，后续版本再做智能影响评估。

### R-09 采购流程（完整流程）

**业务价值**：高。V1 仅有采购建议→审批→订单跟踪，缺少来料质检和入库后的库存更新链路。这是采购数据最后一公里。
**技术复杂度**：中高。需增加：
  - 来料质检单（inspection_records 关联 purchase_orders）
  - 质检通过→自动触发入库（inventory 更新）
  - 部分入库（到货数量 ≠ 采购数量时的处理）
  - 采购完结条件（全量入库 或 手动关闭）
**依赖关系**：依赖 V1 采购订单模块；V1 已有 delivery_notes、purchase_receipts 表结构，数据模型基础完备。
**风险**：质检不合格品的处置流程（退货、降级使用）需提前定义，避免库存数据不一致。

### R-10 销售→生产数据链路

**业务价值**：极高。这是整个系统最核心的链路——打通后，系统才具备"接单→生产→交货"的全程追踪能力，老板才能实时知道每笔订单进度。
**技术复杂度**：极高。涉及：
  - 销售订单确认 → 触发生产工单创建
  - 生产工单 → BOM 展开 → 原材料需求计划
  - 生产排产（工序任务分配给工人）
  - 工人报工（Web端 + 小程序端）
  - 工序完工 → 半成品入库 → 下道工序解锁
  - 成品完工 → 交付确认
**依赖关系**：强依赖 R-04（BOM通用化+版本化）、R-06（Web任务管理）、R-07（客户管理）、R-08（销售订单）。
**风险**：高。链路最长、状态机最复杂，任何中间节点错误均会导致数据不一致；需设计完善的状态机和回滚机制。

### R-11 采购数据链路

**业务价值**：极高。与 R-10 并列，是资金管控的闭环核心。采购建议→审批→订单→质检→入库→库存更新全链路打通后，老板才能真正掌控资金流出节点。
**技术复杂度**：高。相比 R-09 增加了与 R-10 的联动（生产缺料 → 触发采购建议），以及库存更新的准确性要求。
**依赖关系**：依赖 R-09（采购流程）、R-04（BOM版本化）、R-03（价格批量导入完整价格库）。
**风险**：来料不合格退货场景下的库存和采购订单状态同步；部分到货场景的尾单处理。

### R-12 智能调度

**业务价值**：极高（战略级）。这是智造管家区别于普通管理软件的核心差异化能力——AI 驱动的采购建议 + 生产排产建议，把老板从每日决策中解放出来。
**技术复杂度**：极高。需要：
  - 完整、准确的库存数据（依赖 R-11）
  - 完整的生产进度数据（依赖 R-10）
  - 销售订单优先级和交期数据（依赖 R-08）
  - BOM准确展开（依赖 R-04）
  - 调度算法：库存周转分析、资金占用评估、呆滞预警、产能匹配
  - AI 推理生成采购建议和排产建议（Prompt 工程）
**依赖关系**：依赖 R-04、R-08、R-10、R-11 全部完成。是所有需求中依赖链最长的。
**风险**：数据质量直接决定调度结果质量；需设计清晰的"AI建议 vs 人工调整"边界，避免系统过度自动化引发信任危机。

## 四、迭代规划

### 总体规划原则

1. 先补基础主数据（Sprint 1），确保数据层可信
2. 再打通核心链路中台（Sprint 2：BOM强化 + 销售客户 + 采购流程）
3. 然后贯通全链路（Sprint 3）
4. 最后在可信数据基础上做智能调度（Sprint 4）

---

### Sprint 1：主数据补全与体验优化（2周）

**Sprint 目标**：消除 V1 主数据配置痛点，完成高频投诉修复，让工厂能够独立维护基础数据。

**包含需求**：

| 需求 | 优先级 | 工作量估算 |
|---|---|---|
| R-01 SKU类目自定义配置 | P1 | 3天（后端1天 + 前端2天） |
| R-02 供应商导出 + 绩效对比 | P2 | 4天（后端1天 + 前端3天） |
| R-03 采购价格批量导入 | P1 | 5天（后端2天 + 前端3天） |
| R-05 工序极限工时 + 工价配置 | P1 | 4天（后端1.5天 + 前端2.5天） |
| R-06 Web端任务管理 | P1 | 5天（后端0.5天 + 前端4.5天） |

**关键交付物**：
- [artifact:PRD] Sprint 1 功能详细需求文档
- [artifact:Prototype] 类目配置页、供应商对比页、价格导入向导、工序配置增强页、Web任务页
- 后端 API：类目 CRUD、导出接口、Excel解析导入、工序工价接口、Web任务接口
- 前端实现：CategoryConfigPage.tsx、SupplierPage.tsx（独立完整）、PriceImportPage.tsx、ProcessConfigPage.tsx（独立完整）、TaskPage.tsx

**前置依赖**：
- V1 已有 sku_categories 表、supplier_prices 表、process_configs 表、production_tasks 表
- R-06 Web任务管理依赖 V1 后端任务接口（已有）

**Sprint 1 风险**：
- R-03 Excel 解析需设计容错规则（脏数据、格式不一致），需在设计阶段明确校验规则
- R-06 双端（Web + 小程序）并发操作的数据一致性需设计乐观锁

---

### Sprint 2：核心数据中台强化（3周）

**Sprint 目标**：修复 V1 BOM 数据模型核心缺陷，完成客户管理和增强版销售订单，为全链路贯通做数据中台准备。

**包含需求**：

```
As a 工厂老板（系统管理员角色）

- AC3：可在一级类目下新增二级类目，同一一级类目下二级名称唯一
- AC4：可修改类目名称，修改后历史 SKU 的分类归属自动更新显示新名称

**US-V2-006 BOM 版本管理**

```
As a 工厂老板 / 车间主管
I want 为成品或半成品 BOM 创建新版本，并指定某版本为生效版本

- AC7：BOM 版本操作记录可查（版本号、创建人、激活时间、操作备注）
- **优先级**：P0

## 七、技术风险与产品建议

### 风险 T-V2-001：BOM 数据模型变更导致计算引擎回归风险

**风险等级**：P0 / 高
**描述**：R-04 引入 BOM 通用件和版本化后，V1 采购需求计算引擎（suggestion.service.ts）需要大幅改造。现有 20 条 suggestionEngine 测试用例覆盖的是单版本单 BOM 场景，改造后存在回归漏洞风险。
**产品建议**：
1. 要求 Sprint 2 中 BOM 变更单独作为一个可测试里程碑，QA 专项回归时间不少于 2 天
2. 保留 V1 BOM 展开逻辑作为回归对比基准，新旧引擎对同一组测试数据的计算结果须完全一致
3. 上线前需在测试环境导入真实生产数据（脱敏）做端到端验证

### 风险 T-V2-002：生产链路状态机复杂度超预期

**风险等级**：P0 / 高
**描述**：R-10 销售→生产链路涉及多个状态节点（订单确认→工单创建→工序分配→报工→半成品入库→完工→交付），任意节点的异常处理（工序回退、工单取消、插单重排）都需要完整的状态机设计。
**产品建议**：
1. Sprint 3 开始前，tech-lead-architect 必须产出完整的生产状态机 [artifact:架构设计] 文档，经 system-designer 评审通过后才能开始编码
2. V2 阶段暂不支持工序回退（只允许前进状态流转），回退需求延至 V3
3. 工单取消逻辑：仅在"待排产"状态的工单允许取消，进行中的工单只能"异常中止"并记录原因

### 风险 T-V2-003：Excel 批量导入数据质量问题

**风险等级**：P1 / 中
**描述**：R-03 价格批量导入时，工厂历史 Excel 数据格式混乱（合并单元格、多余空行、编码不一致）极可能导致大量解析失败或导入错误数据。
**产品建议**：
1. 提供严格格式的导入模板，模板内嵌数据验证规则（Excel 数据有效性校验）
2. 上传预览阶段必须完整展示解析结果，用户确认后才触发实际导入
3. 提供失败明细下载，帮助用户修正后重新上传
4. 服务端对价格字段做范围合法性校验（不允许负数价格、异常高价需警告但不阻断）

### 风险 T-V2-004：智能调度数据冷启动问题

**风险等级**：P1 / 中
**描述**：R-12 智能调度依赖足够的历史数据（库存周转记录、生产进度数据、采购价格历史）才能产生有价值的建议。V2 上线初期，工厂数据积累不足，AI 建议可信度低。
**产品建议**：
1. Sprint 4 的智能调度首先实现"规则引擎版"（基于安全库存阈值、BOM用量、订单交期的确定性计算），AI 大模型版在数据积累 4-8 周后再开启
2. 建议结果页面明确显示"数据依据"（基于多少天/单的历史数据计算）和"置信度"
3. 任何调度建议必须经人工确认才能转为执行单，禁止全自动下单

### 风险 T-V2-005：双端数据一致性（Web + 小程序）

**风险等级**：P1 / 中
**描述**：R-06 Web 端任务管理上线后，同一任务可能被 Web 端和小程序端同时操作，存在并发冲突风险。
**产品建议**：
1. 任务状态变更使用乐观锁（version 字段），后提交方必须刷新并重新提交
2. 在界面显示任务最后更新时间和更新来源（Web/小程序），帮助用户感知状态

## 十、业务决策确认记录

**记录日期**：2026-03-12
**记录人**：@senior-ai-agent-pm
**决策背景**：Sprint 1 启动前，产品侧与业务方就若干存在歧义的业务规则进行了正式确认，以下决策作为所有后续设计与实现的不可逾越的业务基准。

---

### 总原则

业务底层数据流向和资金流向必须形成闭环。所有模块设计必须围绕这一原则展开。

任何功能设计，若导致数据链断裂或资金流向无法追溯，均视为不符合产品基准，须在评审阶段驳回并重新设计。

---

### 已确认决策（2026-03-12）

#### BD-001 BOM 版本激活规则（快照机制）

**决策**：进行中的生产工单继续沿用创建时锁定的旧 BOM 版本；新接销售订单触发创建的生产工单，使用创建时刻的当前激活 BOM 版本快照。

**影响范围**：
- R-04 BOM版本迭代设计：生产工单创建时必须将激活 BOM 版本号快照写入工单记录（production_orders.bom_version_id 字段必填）
- 计算引擎：BOM展开计算必须以工单关联的版本快照为准，不得动态取当前激活版本
- 测试要求：必须覆盖"工单执行期间 BOM 版本变更，工单计算结果不变"的回归场景

**不允许的设计**：工单创建后，若 BOM 被激活为新版本，工单自动重新计算。

---

#### BD-002 工价计算区分工人等级

**决策**：工人分为熟练工和学徒工两个等级；同一道工序对两种等级设置不同的计件单价；完工上报时，系统根据执行该任务的工人等级自动匹配对应单价计算工资。

**影响范围**：
- R-05 工序工价设计：工序配置表（process_configs）必须支持按等级配置单价，至少设两组字段：unit_price_skilled（熟练工单价）、unit_price_apprentice（学徒工单价）
- 用户主数据：user 表（或 worker_profiles 表）必须包含 skill_level 字段，枚举值：skilled / apprentice
- 工资核算逻辑：计算公式为 完成件数 × 对应等级单价；若工人等级未配置，系统拒绝提交完工上报并提示管理员补全工人信息
- 工资报表：汇总视图需按等级维度分组展示，便于管理层对比熟练工与学徒工的产出成本差异

**不允许的设计**：全工序统一一个计件单价，不区分工人等级。

---

#### BD-003 紧急插单权限

**决策**：紧急插单必须经过具备 admin 角色的公司老板审核确认后才能正式创建对应生产工单；车间主管无权独立确认插单；销售人员仅可提交插单申请。

**权限矩阵**：

| 角色 | 提交插单申请 | 查看影响评估 | 确认插单 |
|---|---|---|---|
| 销售人员（sales） | 允许 | 允许（只读） | 不允许 |
| 车间主管（supervisor） | 不允许 | 允许（只读） | 不允许 |
| 公司老板（admin） | 允许 | 允许 | 允许 |

**影响范围**：
- R-08 销售订单插单流程：后端接口 `POST /orders/:id/confirm-urgent` 必须校验请求方角色为 admin，否则返回 403
- 前端：插单确认按钮仅对 admin 角色渲染，其他角色该按钮不可见
- 通知机制：插单申请提交后，系统必须向所有 admin 角色用户推送待确认通知（站内 + 小程序）
- 验收标准：QA 必须覆盖"车间主管尝试确认插单"被拒绝的负向测试场景

**不允许的设计**：车间主管具有插单确认权限，或插单无需审批直接生效。

---

#### BD-004 来料质检不合格处理规则

**决策**：来料质检结果为不合格时，不允许降级使用（即不合格料不得流入库存），只能执行退货处理；退货后对应采购订单的未入库数量保持不变，系统自动生成退货单并通知采购员处理后续补货。

**影响范围**：
- R-09 来料质检流程：质检结果枚举值为 qualified（合格）/ rejected（不合格），不设"降级使用"选项
- 库存逻辑：仅当质检结果为 qualified 时，系统触发入库操作（inventory 数量增加）；rejected 状态下入库触发接口必须在后端强制拦截，返回业务错误
- 退货单：质检标记为 rejected 后，系统自动创建 return_orders 记录，关联原采购订单和质检单，状态为"待退货"
- 采购订单状态：退货后该批次到货数量回滚，采购订单重新进入"部分到货"或"待到货"状态，等待供应商补发
- 通知：退货事件触发后，自动通知对应采购员和供应商联系人（站内通知）

**不允许的设计**：提供"降级使用"选项将不合格料入库，或不合格质检允许静默忽略直接关闭。

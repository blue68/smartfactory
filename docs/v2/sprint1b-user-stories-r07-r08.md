## 一、Sprint 1b 目标与范围

### 1.1 调度背景

原迭代规划中，R-07（客户管理）和 R-08（销售订单+插单）列于 Sprint 2。业务方确认，若两项功能不尽早就绪，Sprint 3 全链路贯通（R-10 销售→生产链路）将无法按期启动，导致整体延期风险。

因此，将 R-07 和 R-08 调整为与 Sprint 1 **并行开发**，独立为 Sprint 1b 交付轨道，不阻塞 Sprint 1 主线的主数据补全工作。

### 1.2 Sprint 1b 范围

| 需求 | 优先级 | 估算工作量 |
|---|---|---|
| R-07 销售客户管理（含联系人、客户等级、历史订单关联） | P1 | 4天（后端 1.5天 + 前端 2.5天） |
| R-08 销售订单常规流程 + 紧急插单审批流程 | P1 | 6天（后端 2.5天 + 前端 3.5天） |

### 1.3 前置依赖确认

- R-07 是 R-08 的硬性前置：客户主数据必须先就绪，订单才能关联客户。
- R-07 和 R-08 共同是 Sprint 3 中 R-10（销售→生产数据链路）的入口依赖。
- V1 已有 `sales_orders` 表和 `customers` 表的基础结构，Sprint 1b 在此基础上扩展，向前兼容。

## 二、R-07 销售客户管理

### 2.1 US-R07-001 客户档案基础管理

```
As a 销售人员（sales 角色）
I want 在系统中创建、查看、编辑客户基本档案，包含客户编码、名称、联系信息、信用额度和账期天数
So that 录入销售订单时能快速关联已有客户，不再依赖线下表格管理客户信息
```

**优先级**：P1

**验收条件**：

- AC1：侧边导航新增"销售管理 > 客户管理"菜单项，入口对 sales / supervisor / admin 角色可见，对 worker（生产工人）角色不可见。
- AC2：客户列表页展示字段：客户编码、客户名称、客户等级（VIP/A/B/C）、主要联系人姓名及电话、客户状态（活跃/停用）、创建日期。
- AC3：列表支持按客户名称（模糊搜索）、客户编码（精确/模糊）、客户等级、客户状态进行筛选，筛选条件可组合叠加。
- AC4：新增客户表单必填字段：客户名称（不超过 50 字，同一租户下唯一）；系统自动生成客户编码（规则：C + 6位数字序号，如 C000001），不允许手动修改。
- AC5：新增客户表单选填字段：客户等级（VIP/A/B/C，默认 C）、联系地址、邮箱、信用额度（元，精度到分，最小值 0）、账期天数（整数，范围 0-365）、备注。
- AC6：编辑客户时，客户编码不可修改；客户名称若已关联历史订单，修改名称后历史订单中的客户名称显示自动更新（关联显示，非冗余存储）。
- AC7：客户状态可在"活跃"和"停用"间切换；停用状态的客户不出现在订单录入的客户下拉选择列表中，但可在客户管理列表中查看（通过状态筛选）。
- AC8：删除客户时，若该客户有任何历史订单（无论订单状态），系统禁止删除，弹出提示"该客户有 X 条历史订单，无法删除。如需停止与该客户的合作，请将客户状态设为停用。"
- AC9：所有客户的新增、编辑、状态变更操作均记录操作日志（操作人、时间、操作类型、变更字段和前后值）。

### 2.2 US-R07-002 客户联系人管理

```
As a 销售人员（sales 角色）
I want 为一个客户维护多位联系人，并标记主要联系人
So that 跟进不同业务（如下单对接、财务对账、物流收货）时能准确联系到对应负责人
```

**优先级**：P1

**验收条件**：

- AC1：客户详情页包含"联系人"子模块，展示该客户的所有联系人列表：姓名、职位、手机号、邮箱、是否主要联系人标记。
- AC2：新增联系人必填字段：姓名（不超过 20 字）、手机号（11位中国大陆手机号格式校验）；选填字段：职位（不超过 30 字）、邮箱（格式校验）、备注。
- AC3：每个客户有且只有一位"主要联系人"；新增联系人时可勾选"设为主要联系人"，勾选后原主要联系人自动降为普通联系人，系统显示提示"原主要联系人 [姓名] 已变更为普通联系人"。
- AC4：客户至少保留一位联系人；若客户只有一位联系人，不允许删除该联系人，提示"客户须至少保留一位联系人，如需删除请先添加新联系人"。
- AC5：删除联系人时，若该联系人是主要联系人，系统弹出二次确认，要求用户先指定新的主要联系人后才能完成删除。
- AC6：客户列表页和客户选择弹窗中，默认展示主要联系人的姓名和手机号。

### 2.3 US-R07-003 客户等级与历史订单查看

```
As a 工厂老板（admin 角色）或销售人员（sales 角色）
I want 查看某客户的历史订单列表和往来汇总数据
So that 评估客户价值、进行客户等级分类，并在沟通时快速掌握历史合作情况
```

**优先级**：P1

**验收条件**：

- AC1：客户详情页包含"历史订单"子模块，分页展示该客户的所有历史订单（20条/页），字段：订单编号、订单日期、交付日期、订单金额、订单状态。
- AC2：历史订单列表支持按订单状态筛选（草稿/已确认/生产中/已发货/已完成/已关闭）、按订单日期范围筛选。
- AC3：历史订单区域展示汇总数据：历史订单总数量、历史订单总金额（元，保留两位小数）、最近下单日期。
- AC4：点击订单编号可跳转至对应订单详情页（跨模块跳转）。
- AC5：客户等级字段（VIP/A/B/C）可在客户详情页和客户列表页直接编辑（inline edit），变更后立即保存并记录操作日志；等级变更不影响历史订单数据。
- AC6：客户列表支持按客户等级快速筛选，方便销售经理批量管理不同等级客户。

### 2.4 US-R07-004 客户数据导出

```
As a 工厂老板（admin 角色）或销售人员（sales 角色）
I want 将客户列表导出为 Excel 文件
So that 用于外部汇报、数据备份或在线下会议中使用
```

**优先级**：P2

**验收条件**：

- AC1：客户列表页提供"导出"按钮，点击后导出当前筛选条件下的全部客户数据（不限分页）。
- AC2：导出 Excel 包含字段：客户编码、客户名称、客户等级、客户状态、联系地址、主要联系人姓名、主要联系人手机号、主要联系人邮箱、信用额度、账期天数、历史订单数量、历史订单总金额、创建日期。
- AC3：导出文件命名规则：`客户列表_YYYYMMDD_HHMMSS.xlsx`。
- AC4：导出条数超过 1000 条时，系统显示加载中提示，导出完成后自动触发浏览器下载，不阻塞页面其他操作。
- AC5：导出不包含客户的完整联系人列表（主要联系人单条信息已足够），如需完整联系人数据需进入客户详情页手动查看。

## 三、R-08 销售订单 + 紧急插单

### 3.1 US-R08-001 常规销售订单创建与管理

```
As a 销售人员（sales 角色）
I want 创建销售订单，填写客户、产品、数量、单价和交付日期，并追踪订单状态
So that 准确记录每一笔业务成交，并能实时了解订单从确认到交付的完整进度
```

**优先级**：P1

**验收条件**：

- AC1：销售订单新建表单包含订单头信息：关联客户（从已有活跃客户中选择，必填）、订单日期（默认今天，必填）、要求交付日期（必填，不得早于订单日期）、订单优先级（普通/紧急，默认普通）、备注（选填）。
- AC2：订单明细行（至少 1 行）：产品 SKU（从已有成品 SKU 中选择，必填）、数量（正整数，必填）、单价（元，精度到分，必填）、金额（系统自动计算：数量 × 单价，只读）。
- AC3：支持动态增减明细行：点击"添加明细"追加新行，点击行末"删除"图标移除该行，删除前最后一行时提示"订单须至少保留一条明细"。
- AC4：订单金额汇总：页面底部实时显示订单总金额（所有明细行金额之和）。
- AC5：订单编号系统自动生成，规则：SO + 8位日期（YYYYMMDD）+ 4位流水号（如 SO202603120001），不允许手动修改。
- AC6：保存为草稿状态：填写不完整时可点击"保存草稿"，草稿订单可继续编辑，草稿状态不触发任何下游流程。
- AC7：订单状态流转如下，仅允许按序前进（V2 阶段不支持回退）：

```
草稿 → 已确认 → 生产中 → 已发货 → 已完成 → 已关闭
```

- AC8：状态操作规则：
  - 草稿 → 已确认：由销售人员点击"确认订单"触发，确认后不允许再修改订单明细（客户、产品、数量、单价均锁定）；普通订单确认后直接进入"已确认"，无需审批。
  - 已确认 → 生产中：由后端在生产工单创建成功后自动触发（非手动操作）。
  - 生产中 → 已发货：由车间主管或 admin 手动点击"标记发货"触发，需填写实际发货日期和物流单号（选填）。
  - 已发货 → 已完成：由 admin 或 sales 手动点击"确认收货"触发，或在发货后 7 天无操作时系统自动流转。
  - 已关闭：任意状态下，admin 可手动关闭订单，需填写关闭原因（必填）；生产中状态的订单关闭需额外确认（提示"该订单已有关联生产工单，关闭后工单将标记为异常中止"）。
- AC9：订单列表页展示：订单编号、客户名称、订单日期、要求交付日期、订单优先级（紧急订单红色标签高亮）、订单总金额、订单状态。列表默认按订单日期倒序排列。
- AC10：订单列表支持筛选：按客户名称（模糊）、订单状态、订单日期范围、订单优先级（普通/紧急）。

### 3.2 US-R08-002 紧急插单提交

```
As a 销售人员（sales 角色）
I want 将订单标记为"紧急"并提交插单申请，并在提交前看到系统给出的产能影响提示
So that 接到客户紧急需求时，能快速记录并将影响告知老板，避免盲目承诺交期
```

**优先级**：P0（业务决策 BD-003 约束项）

**验收条件**：

- AC1：订单表单中"订单优先级"字段选择"紧急"后，页面展示黄色信息提示区，实时调用产能查询接口，显示："系统当前生产负荷状态正在计算中..."（加载态）。
- AC2：产能查询接口返回后，提示区展示以下信息（简化版，V2 阶段基于优先级插队逻辑估算）：
  - 当前最早可排产日期：YYYY-MM-DD
  - 预计最早完工日期：YYYY-MM-DD（基于排产日期 + 产品标准工时估算）
  - 若客户要求交付日期早于预计最早完工日期：提示区变为红色，显示"交期紧张！按当前排产，预计无法按期完成。接单后可能影响以下 X 个在产工单的交期：[工单编号列表，最多展示 5 条，超出显示'等 X 项']（预计延期约 N 天）"
- AC3：产能查询接口调用超时（超过 5 秒）时，提示区显示"产能状态查询超时，请手动评估影响后再提交"，不阻断订单提交流程。
- AC4：紧急订单填写完成后，点击"提交插单申请"（非"确认订单"）：订单状态变为"待审批"，不可再编辑订单内容；同时向所有 admin 角色用户推送站内通知和小程序消息通知，通知内容包含：订单编号、客户名称、订单金额、要求交付日期、产能影响提示摘要。
- AC5：紧急订单在订单列表中以红色"紧急"标签高亮显示，"待审批"状态以橙色标签显示，与普通订单在视觉上明显区分。
- AC6：销售人员在订单详情页可查看当前审批状态（待审批/已通过/已驳回）及审批备注，但不可操作审批动作。
- AC7：销售人员可撤回"待审批"状态的插单申请（在 admin 尚未操作之前），撤回后订单状态回到"草稿"，可重新编辑后再次提交。

### 3.3 US-R08-003 紧急插单审批（老板审核）

```
As a 工厂老板（admin 角色）
I want 查看待审批的紧急插单申请，了解影响评估详情，并做出通过或驳回决策
So that 在知情的情况下决策是否接受紧急订单，避免因插单导致已有订单意外延期且无人知晓
```

**优先级**：P0（业务决策 BD-003 约束项）

**验收条件**：

- AC1：导航顶部及订单管理入口展示待审批紧急插单数量的红点角标（实时更新），有新插单申请时触发更新。
- AC2：订单管理页面提供"紧急插单审批"专属视图（Tab 或独立筛选），列出所有"待审批"状态的插单，按申请时间倒序排列。
- AC3：点击插单申请进入审批详情页，展示：
  - 订单基本信息：客户、产品明细、数量、金额、要求交付日期
  - 产能影响评估：最早可排产日期、预计完工日期、受影响在产工单列表（工单编号、原预计完工日期、预计延期天数）
  - 申请人信息：销售人员姓名和提交时间
- AC4：admin 点击"通过审批"按钮，需填写审批备注（选填），确认后订单状态变为"已确认"，系统自动触发生产工单创建流程（通知排产模块），同时通知申请销售人员"插单申请已通过"。
- AC5：admin 点击"驳回申请"按钮，必须填写驳回原因（必填，不超过 200 字），确认后订单状态变为"已驳回"，通知申请销售人员，销售人员可在驳回详情中查看驳回原因。
- AC6：已驳回的订单可由销售人员"重新编辑后再次提交"（修改交付日期或数量后重新发起），重新提交后状态回到"待审批"，重新通知所有 admin 用户；单个订单最多允许重新提交 3 次，超出后订单自动关闭并提示"已超出最大重新提交次数，请创建新订单"。
- AC7：审批操作（通过/驳回）必须记录审计日志：操作人（admin 姓名）、操作时间、审批结果、审批备注/驳回原因。
- AC8：**权限强制校验**：`POST /orders/:id/confirm-urgent` 接口后端必须校验请求用户角色为 admin；非 admin 角色调用该接口返回 HTTP 403，响应体 `{code: 403, message: "无权操作：紧急插单审批仅限 admin 角色执行"}`；前端审批确认按钮仅对 admin 角色渲染，其他角色不渲染该按钮（不依赖 disabled 状态，直接不渲染）。

### 3.4 US-R08-004 订单详情与状态追踪

```
As a 工厂老板（admin 角色）或销售人员（sales 角色）
I want 在订单详情页看到订单的完整状态时间线和关联生产工单信息
So that 随时掌握订单从录入到交付的全程进度，在客户询问时能准确回答
```

**优先级**：P1

**验收条件**：

- AC1：订单详情页顶部展示订单基本信息（订单编号、客户、状态、优先级、金额）和状态流转时间线（各状态节点的到达时间，未到达的节点显示为待办状态）。
- AC2：订单明细区展示所有产品明细行，含 SKU 编码、名称、数量、单价、金额，以及底部总金额汇总。
- AC3：若订单已进入"生产中"状态，展示关联生产工单区域：工单编号、工单状态、创建时间、预计完工日期（只读，由排产模块填充）。
- AC4：若为紧急插单，展示审批记录区：审批状态、审批人、审批时间、审批备注（或驳回原因）。
- AC5：订单操作按钮区根据当前状态和当前用户角色动态展示可操作按钮（不可操作的按钮不渲染）：
  - 草稿：sales/admin 可见"保存草稿"、"确认订单"（普通）或"提交插单申请"（紧急）
  - 待审批：admin 可见"通过审批"、"驳回申请"；sales 可见"撤回申请"
  - 已确认/生产中：supervisor/admin 可见"标记发货"；admin 可见"关闭订单"
  - 已发货：sales/admin 可见"确认收货"；admin 可见"关闭订单"
- AC6：操作日志区展示该订单的全部状态变更记录（操作人、时间、操作类型、备注）。

## 四、数据流说明

### 4.1 销售订单触发生产排产的完整数据链路

以下为 R-07/R-08 在整体数据链路中的位置和触发关系（当前 Sprint 1b 实现前两个节点，后续节点由 Sprint 3 R-10 完成）：

```
[节点 1] 销售人员创建客户档案（R-07）
    客户数据写入 customers 表（customer_id, name, grade, credit_limit, payment_days...）
    ↓

[节点 2A] 销售人员创建普通订单（R-08）
    订单数据写入 sales_orders 表（priority = 'normal'）
    订单明细写入 sales_order_items 表（sku_id, quantity, unit_price）
    → 销售人员点击"确认订单"
    → 订单状态变更：草稿 → 已确认
    → 触发事件：ORDER_CONFIRMED（事件发布至消息队列，Sprint 3 R-10 消费此事件创建生产工单）

[节点 2B] 销售人员创建紧急订单（R-08）
    订单数据写入 sales_orders 表（priority = 'urgent'，status = 'pending_approval'）
    → 通知推送至所有 admin 用户
    → admin 审批通过：订单状态变更为"已确认"
    → 触发事件：ORDER_CONFIRMED（同上，生产优先级更高）
    → admin 审批驳回：订单状态变更为"已驳回"，无后续事件

[节点 3] 生产排产（Sprint 3 R-10 实现）
    生产模块消费 ORDER_CONFIRMED 事件
    → 读取订单明细中的 sku_id 和 quantity
    → 读取对应成品 SKU 的当前激活 BOM 版本快照（BD-001 决策）
    → 创建生产工单（production_orders 表），锁定 bom_version_id
    → 生产工单状态：待排产 → 排产中 → 生产中
    → 销售订单状态联动更新：已确认 → 生产中

[节点 4] 报工与完工（Sprint 3 R-10 实现）
    工人在 Web 端或小程序端完工上报（R-06）
    → 工序任务状态更新
    → 所有工序完成 → 生产工单完工
    → 成品进入待发货库存

[节点 5] 发货与交付（Sprint 3 R-10 实现）
    车间主管/admin 标记发货 → 销售订单状态：生产中 → 已发货
    销售/admin 确认收货 → 销售订单状态：已发货 → 已完成
    资金流记录：应收账款金额 = 订单总金额，账期 = 客户账期天数
```

### 4.2 数据完整性约束（Sprint 1b 范围内）

| 约束点 | 规则 |
|---|---|
| 订单确认后明细不可变 | 已确认/生产中/后续状态的订单，sku_id、quantity、unit_price 字段锁定，后端禁止 UPDATE 操作 |
| 客户不可删除（有订单） | 后端在 DELETE /customers/:id 接口中强制校验是否有关联订单，有则返回业务错误码 |
| 紧急插单状态机完整性 | pending_approval 状态仅允许 admin 操作（approve/reject）或申请者操作（withdraw）；其他角色的状态变更操作后端拒绝 |
| 订单金额精度 | 所有金额字段使用 DECIMAL(15,2) 类型存储；前端展示保留两位小数；金额计算在后端执行，前端展示只读结果 |

|---|---|

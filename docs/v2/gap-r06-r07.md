## R-06 Web端任务管理

### 设计稿 vs 前端实现

#### 🔴 R06-G01：统计卡片数据来源错误，缺少独立统计接口

**严重度**：重大缺失

**设计稿 / 交互说明要求**：
- 统计卡片展示「待开始、进行中、已完成、异常待处理」四类计数
- 交互说明明确：数字来自 `GET /production-tasks/stats` 独立接口
- 卡片数字在页面加载后执行 countUp 动画（从 0 到实际值，600ms ease-out）

**前端实现现状**：
```
// TaskPage.tsx L235-241
const stats = {
  total:      taskData.total,
  inProgress: taskData.list.filter((t) => t.status === 'in_progress').length,
  completed:  taskData.list.filter((t) => t.status === 'completed').length,
  exception:  taskData.list.filter((t) => t.status === 'exception').length,
};
```
统计数字从当前分页列表（pageSize=20）的本地 filter 计算得出，不代表全库数量。`total` 来自分页 total 字段，但 inProgress/completed/exception 只计算当前页 20 条的数据。

**后端现状**：`production.service.ts` 无 `getStats` / `getDashboardStats` 方法，`GET /production-tasks/stats` 接口不存在。

**影响**：统计卡片数值与实际全局数量不符，误导用户决策；大数据量下偏差更严重。

#### 🔴 R06-G02：完工上报弹窗缺少实际工时字段和工资预览

**严重度**：重大缺失

**设计稿要求**：
- 完工上报弹窗包含两个必填字段：**完成件数** + **实际工时（小时）**
- 弹窗内有工资预览卡片：展示工人等级、计算公式（件数 × 单价）、预计工资金额（实时 countUp 更新）
- 实际工时输入后触发超时预警检测：人均工时 ≥ 极限工时 × 1.2 时显示黄色预警横幅
- 工人等级未配置时弹窗置灰并显示红色错误横幅

**前端实现现状**：
- `CompleteTaskModal` 仅包含「完成数量」和「备注」两个字段，无实际工时输入
- 无工资预览区域
- 无超时预警检测逻辑
- 无工人等级未配置的错误处理

**后端实现现状**：
- `scheduler.completeTask` 方法签名包含 `scrapQty`、`scrapReason`、`componentBarcode`、`images` 等字段，但未接收 `actualHours`（实际工时）
- `listTasks` 返回的列表数据不含 `unit_price`（计件单价）、`worker_grade`（工人等级）

**影响**：工资核算的核心流程完全缺失，超时预警无法触发，工资计算无数据来源。

#### 🔴 R06-G03：异常上报缺少「是否影响生产进度」字段和图片上传

**严重度**：重大缺失

**设计稿要求**：
- 异常类型选择：卡片式单选（设备故障/物料缺失/质量问题/其他），每项含 Tooltip 帮助说明
- 必填字段「是否影响生产进度」（是/否 二选一按钮）
- 图片上传区：拖拽/点击上传，最多 3 张，单张 ≤5MB，JPG/PNG，含上传进度条
- 描述字数最少 10 字校验

**前端实现现状**：
- 异常类型改为 `<select>` 下拉而非卡片式单选（交互体验不符）
- 增加了设计稿未要求的「严重程度」（low/medium/high）下拉，与设计脱节
- 无「是否影响生产进度」字段
- 无图片上传功能
- 描述字数校验仅检查非空，未校验 10 字最小长度

**后端实现现状**：
- `reportException` 接收 `{ type, description, severity }` 三字段
- 无 `affectsProgress`（影响生产进度）字段
- 无图片 URL 数组字段
- severity（严重程度）是前端自创字段，设计稿无此概念

#### 🔴 R06-G04：任务详情抽屉缺少 BOM 快照区和操作时间线

**严重度**：重大缺失

**设计稿 / 交互说明要求**：
- 抽屉包含四个区域：基本信息、BOM 快照（折叠展开，只读表格）、操作时间线、操作按钮区
- BOM 快照显示：物料名称、SKU编码、用量、单位
- 操作时间线：每条含操作类型图标（彩色圆点）、操作人、时间、描述；初始显示最新5条，超出显示「查看全部」
- 详情中展示计件单价、极限工时

**前端实现现状**：
- `TaskDetailContent` 包含：基本信息、生产数据、SKU信息、异常记录
- **无 BOM 快照区**
- **无操作时间线**（仅有异常记录列表，非完整时间线）
- 详情中无计件单价、极限工时字段

**后端实现现状**：
- `getTaskDetail` 返回数据含 `standardHours`、`maxHours`，但无 BOM 数据
- 无 `GET /production-tasks/{id}/logs` 接口和对应 service 方法
- 无任务操作日志表的查询逻辑

#### 🔴 R06-G05：缺失「已挂起」状态和主管异常处置流程

**严重度**：重大缺失

**设计稿 / 交互说明要求**：
- 任务状态机包含：待开始 → 进行中 → 已完成 / 异常待处理 → 进行中（标记处理）/ 已挂起
- 主管/管理员视角：异常任务抽屉底部显示「标记已处理」和「挂起任务」两个专属按钮
- 「标记已处理」：弹窗填写处理说明，提交后状态回到「进行中」
- 「挂起任务」：弹窗填写挂起原因（必填），提交后状态变为「已挂起」（橙色）

**前端实现现状**：
- `TaskStatus` 类型定义：`'pending' | 'in_progress' | 'completed' | 'exception'`，无 `suspended`
- `TaskDrawerFooter`：exception 状态下只有「处理异常」按钮（触发 openExceptionModal），逻辑为「再次上报异常」而非「主管处置」
- 无「挂起任务」弹窗、无「标记已处理」弹窗、无权限判断逻辑

**后端实现现状**：
- `resolveException` 方法将任务状态恢复为 `pending` 而非 `in_progress`（与设计要求不符）
- 无 `suspendTask` 方法，无 `suspended` 状态写入逻辑

## R-07 客户管理

### 设计稿 vs 前端实现

#### 🔴 R07-G01：新增客户表单与设计稿字段结构不符

**严重度**：重大缺失

**设计稿要求（新增客户 Modal）**：
- 分三个 section：基本信息（客户名称、客户编码[只读/系统生成]、客户等级、所在地区、详细地址）+ 主要联系人（姓名、职位、手机、邮箱）+ 财务信息（信用额度、账期）+ 备注
- 客户编码为**只读**，系统自动生成（`form-input--readonly`）
- 主要联系人在新增时作为表单内嵌 section，包含职位字段

**前端实现现状**：
- 客户编码为**可编辑**输入框（无 readonly），需用户手动输入
- 主要联系人字段：仅有「联系人姓名」和「电话」，无职位字段
- 无分 section 的 form 结构（设计稿有 form-section__title 分组）
- 字段排列为平铺 grid，无层次区分

---

#### 🔴 R07-G02：导出功能缺失

**严重度**：重大缺失

**设计稿**：页头 actions 区有「导出」按钮（导出 Excel）

**前端现状**：页头只有「新增客户」按钮，无导出入口。

**后端现状**：`sales-customer/customer.service.ts` 和 `customer/customer.service.ts` 均无导出相关方法。

---

#### 🔴 R07-G03：停用客户操作错误处理不完整

**严重度**：重大缺失

**设计稿 / 业务逻辑要求**：停用有活跃订单的客户，后端返回 400 + 活跃订单数量，前端需展示具体错误信息给用户

**前端现状**：
```
// CustomerPage.tsx L599-604
} catch {
  // API will return error with active order count — handled by global error handler
}
```
catch 块为空，仅注释说明依赖全局错误处理。用户无法得知是因为「有 X 个活跃订单」导致停用失败，体验极差。

**后端现状**：`updateStatus` 逻辑正确，返回 `该客户有 ${activeOrderCount} 个进行中的订单，无法停用`，但前端丢弃了此信息。

---

#### 🟡 R07-G04：详情 Drawer 结构与设计稿存在布局差异

**严重度**：部分缺失

**设计稿要求**：
- Drawer 头部：客户名称（大标题）+ 等级 Tag 横排，副标题显示「编码：XXX · 创建于 YYYY-MM-DD」
- 基本信息使用 `info-card` 卡片样式（灰色背景 + 圆角边框包裹）
- 联系人卡片有**头像首字母圆形**（`contact-item__avatar`）
- Drawer 底部 footer 有「关闭」+「编辑客户」两个按钮

**前端现状**：
- Drawer title 仅为 `${customer.name} — 客户详情`，无等级 Tag 横排，无创建时间副标题
- 基本信息为普通 div 列表，无 info-card 卡片包裹样式
- 联系人卡片无头像首字母圆形组件
- Drawer 无底部 footer 操作按钮（无「编辑客户」快捷入口）

---

#### 🟡 R07-G05：历史订单列表状态显示不规范

**严重度**：部分缺失

**设计稿要求**：历史订单卡片显示订单号、日期、交付日、金额、状态（含彩色点状 badge，如「生产中」为橙色）

**前端现状**：
```
// CustomerPage.tsx L507-510
<span className={`${styles.statusBadge} ${

  {o.status}  // 直接显示英文枚举值
```

- 状态展示英文原始值（如 `in_production`），未做中文映射
- 只有两种样式（active/inactive），无「生产中」橙色等中间状态样式
- 订单卡片无交付日期字段展示

#### 🟡 R07-G06：筛选栏缺少「共 N 条」计数显示

**严重度**：细节差异

**设计稿**：筛选工具栏右侧显示「共 128 条」实时计数

**前端现状**：计数信息在分页区域（`共 {data?.total} 条`），仅在超过1页时才显示分页组件，筛选栏内无计数。

---

#### 🟡 R07-G07：两个 customer.service.ts 共存，模块边界混乱

**严重度**：部分缺失

**现状**：项目中存在两个 CustomerService 实现：
- `/modules/customer/customer.service.ts`：简版，无等级筛选、无联系人 CRUD、无订单关联、`CreateCustomerParams` 仅有4字段
- `/modules/sales-customer/customer.service.ts`：完整版，功能齐全

前端 API 文件引用的是哪个模块的路由，需确认实际挂载情况。简版模块若被误用，将导致等级筛选、联系人管理、客户订单关联全部失效。

#### 🟢 R07-G08：新增联系人表单缺少邮箱字段

**严重度**：细节差异

**设计稿联系人**：姓名、职位、手机号码、邮箱

**前端 Drawer 内新增联系人表单**：姓名、职位、电话（无邮箱输入框）

**后端 `addContact`**：`CreateContactParams` 包含 `email` 字段，接口支持，仅前端表单未渲染。

---

#### 🟢 R07-G09：主联系人删除保护未在前端给予友好提示

**严重度**：细节差异

**后端逻辑**：主联系人调用删除接口返回 400 + 错误信息"主要联系人不可直接删除，请先将其他联系人设为主要联系人"

**前端现状**：`handleDeleteContact` 无错误处理，catch 未捕获，删除按钮对主联系人同样显示，点击后失败无任何提示反馈。

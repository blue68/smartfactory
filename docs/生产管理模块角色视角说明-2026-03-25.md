# 生产管理模块角色视角说明 - 2026-03-25

## 1. 目标

本文从不同角色的日常使用视角，解释生产管理 5 个子模块的进入顺序、关注点和操作边界。

涉及模块：

1. 智能调度
2. 排产计划
3. 生产任务
4. 生产工单
5. 缺料看板

---

## 2. 总览

不同角色看到的是同一条生产链上的不同切面。

```text
销售订单确认
  -> 生产工单
  -> 缺料检测
  -> 调度建议
  -> 排产计划
  -> 生产任务
  -> 完工 / 入库 / 交付
```

角色分工：

- 销售：关心订单是否顺利转生产、是否影响交付
- 车间主管：关心工单优先级、排产、异常处理
- 工人：关心今天做什么、做到哪里、怎么报工
- 采购：关心哪些工单因缺料卡住、要补什么料

---

## 3. 销售视角

### 3.1 销售最关心什么

销售通常不深度操作生产模块，但需要知道：

1. 客户订单是否已经转成生产工单
2. 当前有没有缺料风险
3. 工单是否已排产、是否开工
4. 是否会影响交期和交付

### 3.2 销售主要看哪些页面

1. 订单管理
2. 订单详情中的生产进度区块
3. 生产工单列表
4. 缺料看板

### 3.3 销售如何理解 5 个模块

#### 生产工单

- 这是销售判断“订单有没有真正进入工厂”的第一入口
- 一张销售订单可能拆成多张生产工单
- 销售主要看：
  - 工单号
  - 产品
  - 计划数量
  - 当前状态
  - 完工进度

#### 缺料看板

- 这是销售识别“为什么交期可能会被拖延”的关键页面
- 如果某个订单对应工单在缺料，销售就知道问题不在车间排产，而在供应链

#### 智能调度

- 销售一般不直接操作
- 但可以把它理解为“系统是否认为这张工单应该优先排”
- 对紧急订单尤其重要

#### 排产计划

- 销售通常只读
- 主要看订单是否已经进入正式排产

#### 生产任务

- 销售一般不进
- 这是执行层，不适合销售日常使用

### 3.4 销售的典型路径

```text
订单管理
  -> 查看订单详情
  -> 查看关联生产工单
  -> 若发现交期风险
  -> 查看缺料看板 / 询问车间主管
```

---

## 4. 车间主管视角

### 4.1 车间主管最关心什么

车间主管是这 5 个模块的核心使用者。

他最关心：

1. 现在有哪些工单待生产
2. 哪些工单更紧急
3. 物料是否齐套
4. 今天怎么排
5. 工人任务执行到哪一步
6. 异常在哪里

### 4.2 车间主管主要看哪些页面

1. 生产工单
2. 缺料看板
3. 智能调度
4. 排产计划
5. 生产任务

### 4.3 车间主管如何理解 5 个模块

#### 生产工单

- 这是主管的“生产池”
- 所有待做的工作都先以工单形式出现
- 主管会在这里看：
  - 哪些工单待排产
  - 哪些工单优先级高
  - 哪些工单已经开工
  - 哪些工单进度落后

#### 缺料看板

- 这是主管判断“能不能排、该不该排”的前置页
- 某张工单如果严重缺料，主管即使想优先排，也可能无法真正执行
- 所以缺料看板相当于排产前的可执行性检查

#### 智能调度

- 这是主管的参考建议页
- 系统会基于：
  - 交期
  - 优先级
  - 齐料率
  - 工人负荷
  给出建议排序
- 主管不用完全照 AI 来，但会用它快速识别优先工单

#### 排产计划

- 这是主管最核心的执行页面
- 在这里完成：
  - 查看当日排产
  - 拖拽调整
  - 查看冲突
  - 确认下发

#### 生产任务

- 这是主管查看落地执行的页面
- 排产一旦确认，关注点就转到任务执行：
  - 是否开始
  - 是否超时
  - 是否有异常
  - 是否需要挂起 / 恢复

### 4.4 车间主管的典型路径

```text
生产工单
  -> 缺料看板
  -> 智能调度
  -> 排产计划
  -> 确认下发
  -> 生产任务
  -> 处理异常 / 跟进进度
```

### 4.5 车间主管最容易混淆的点

1. `智能调度` 不是正式排产
- 它只是建议

2. `排产计划` 和 `生产任务` 不是同一层
- 排产计划是主管安排
- 生产任务是工人执行

## 5. 工人视角

### 5.1 工人最关心什么

工人不关心宏观调度，只关心：

1. 今天轮到我做什么
2. 做哪一个工单的哪道工序
3. 数量是多少
4. 怎么开始
5. 怎么报工
6. 出现异常怎么办

### 5.2 工人主要看哪些页面

1. 生产任务

工人一般不需要主动看：

- 智能调度
- 排产计划
- 缺料看板

### 5.3 工人如何理解 5 个模块

#### 生产工单

- 工人通常不直接操作
- 只会在任务详情里感知到它的上下文，比如：
  - 工单号
  - 产品名
  - 销售订单号

#### 缺料看板

- 工人通常不直接看
- 如果缺料，影响会体现为任务无法正常开始或被挂起

#### 智能调度

- 工人不关心

#### 排产计划

- 工人也不需要直接排
- 它只是主管给工人分配任务前的中间层

#### 生产任务

- 这是工人的主工作台
- 工人的全部动作都在这里发生：
  - 开始任务
  - 上报完成数量
  - 上报报废数量
  - 上报异常

### 5.4 工人的典型路径

```text
查看今日任务
  -> 开始任务
  -> 执行工序
  -> 完工上报
  -> 若异常则上报异常
  -> 等待下道任务或新任务
```

## 6. 采购视角

### 6.1 采购最关心什么

采购不关心具体怎么排班，但非常关心：

1. 哪些工单因为缺料卡住
2. 缺的是哪种物料
3. 缺口有多大
4. 需求日期是什么时候
5. 是否已经生成采购建议

### 6.2 采购主要看哪些页面

1. 缺料看板
2. 采购建议
3. 采购订单
4. 入库记录

在生产这 5 个模块里，采购最常用的是：

1. 缺料看板

### 6.3 采购如何理解 5 个模块

#### 生产工单

- 采购不直接管理工单
- 但采购建议和缺料分析都能追溯到工单

#### 缺料看板

- 这是采购进入生产链路的主入口
- 缺料看板告诉采购：
  - 缺什么
  - 缺多少
  - 影响多少张工单
  - 哪些需求日期最紧

#### 智能调度

- 采购通常只关注里面的采购建议部分
- 对生产排产建议本身关注较少

#### 排产计划

- 采购通常不直接用
- 但会用它判断“哪些缺料真的会马上影响当日排产”

#### 生产任务

- 采购不直接使用

### 6.4 采购的典型路径

```text
缺料看板
  -> 采购建议
  -> 审批 / 下采购单
  -> 到货 / 入库
  -> 缺料状态重评
```

## 8. 不同角色的推荐理解顺序

### 销售

```text
订单管理 -> 生产工单 -> 缺料看板
```

### 车间主管

```text
生产工单 -> 缺料看板 -> 智能调度 -> 排产计划 -> 生产任务
```

### 工人

```text
生产任务
```

### 采购

```text
缺料看板 -> 采购建议 -> 采购订单 -> 入库
```

## 9. 角色视角下的最终结论

同样是这 5 个模块，不同角色其实在用不同的子集：

- 销售把它看成“订单有没有顺利进厂”
- 车间主管把它看成“从待产到执行的完整控制台”
- 工人把它看成“今日任务清单”
- 采购把它看成“缺料和补料入口”

所以你会感觉它们关系复杂，根本原因不是模块设计太多，而是：

- 一个系统同时服务了多个角色
- 每个角色只需要理解其中一部分

如果后续要培训团队，最合理的方式不是按模块讲，而是按角色讲。

# 权限控制模块 PRD

## 1. 入口检查

### 1.1 目标明确性

已明确。本次目标是为现有系统补齐“权限控制模块”，形成从租户到人员、从菜单到功能点、从角色到数据范围的完整可配置能力。

### 1.2 必需输入完整性

已具备最小输入。现有系统已有：

- 登录认证模型
- 固定角色枚举
- Sidebar 菜单定义
- 业务模块页面与角色显隐关系

缺失的是可配置权限中心本身，但不影响产品定义阶段推进。

### 1.3 角色权限检查

符合。当前由 `senior-ai-agent-pm` 承担产品定义职责，不涉及实现代码编写。

### 1.4 前置审批检查

无。产品定义为上游阶段，可直接启动。

## 2. 背景与问题

当前系统已具备多租户基础、登录鉴权能力及固定角色体系：

- 角色枚举：`admin / boss / purchaser / warehouse / supervisor / worker / qc / sales`
- 前端通过 `authStore.hasRole/hasAnyRole` 做按钮显隐
- 侧边栏通过 `Sidebar.tsx` 中的静态 `NAV_ITEMS` 按角色控制可见菜单

但当前方案存在四类问题：

1. 权限不可配置。新增组织、岗位、临时岗位或组合职责时只能改代码。
2. 菜单与功能点未分离。页面可见不代表按钮、导出、审批、转单等操作受控。
3. 人员与角色关系粗糙。无法支持一人多角色、代理授权、禁用角色、按租户独立管理。
4. 租户治理能力不足。缺少租户启停、套餐、功能开关、默认角色模板等平台能力。
5. 系统管理员边界不清。当前登录模型默认依赖 `tenantCode`，无法支撑“平台超级管理员先登录平台，再显式切换到目标租户上下文”的受控代管模式。

因此需要建设一个独立的权限控制模块，支持“系统级模板 + 租户级配置 + 人员分配 + 功能点授权 + 数据范围控制”。

## 3. 产品目标

### 3.1 业务目标

1. 让老板与 `platform_super_admin` 无需改代码即可配置菜单、功能点、角色和人员授权。
2. 让不同租户拥有独立的权限配置，不相互影响。
3. 让现有固定角色体系平滑迁移为可配置 RBAC 体系，避免一次性重构所有业务模块。
4. 为后续移动端、小程序、API 开放平台统一权限校验提供主数据基础。
5. 将平台级管理员定义为显式的 `platform_super_admin`，通过“平台态 + 切换租户上下文”的方式工作，而不是默认无边界跨租户。

### 3.2 成功标准

1. 能在后台创建/禁用租户。
2. 能配置菜单树、功能点、角色、人员及人员角色分配。
3. 能为角色配置菜单可见权限、功能操作权限、数据范围。
4. 能兼容当前固定角色，旧账号不迁移也能继续正常登录使用。
5. 敏感操作可做到“页面不可见 + 接口不可调用”双重控制。
6. `platform_super_admin` 可不输入 `tenantCode` 登录平台控制台，并在审计留痕前提下切换进入目标租户上下文。

## 4. 范围

### 4.1 本期范围

本期建设 PC 端权限控制模块，包含以下六个页面域：

1. 租户配置
2. 系统菜单与功能模块配置
3. 角色配置
4. 人员配置
5. 角色菜单及功能配置
6. 人员角色分配

本期还包含以下规则能力：

- 预置系统角色模板
- 菜单树与功能点树
- 数据范围模型
- 角色禁用/启用
- 人员状态管理
- 租户功能开关
- 兼容现有固定角色的映射机制

### 4.2 非目标

本期不包含：

1. SSO、LDAP、企业微信/飞书组织同步
2. 审批流授权矩阵的细粒度条件表达式引擎
3. 字段级权限、列级脱敏
4. 跨租户共享人员池
5. 移动端权限页面
6. 自定义公式型权限脚本

## 5. 用户与使用场景

### 5.1 核心用户

- 平台超级管理员：`platform_super_admin`，维护租户、系统菜单模板、系统预置角色，并可切换进入目标租户上下文执行系统管理操作
- 租户管理员：维护本租户角色、人员、授权关系
- 工厂老板：查看与调整本租户高权限角色授权
- 业务主管：仅在被授权时可管理本部门人员角色

### 5.2 典型场景

1. 新租户开通时，`platform_super_admin` 选择行业模板，一键生成默认菜单与角色。
2. 某租户新增“销售主管”岗位，需要在不改代码情况下配置销售订单审批权限。
3. 某员工临时兼任“仓管 + 盘点负责人”，需分配两个角色并限定数据范围。
4. 租户暂停 AI 功能，仅关闭 AI 菜单与 AI 功能点，不影响基础业务。
5. 平台超级管理员登录后，需要先选择“进入某租户上下文”，再查看和维护该租户的角色、人员和授权配置。

## 6. 权限模型

### 6.1 模型选择

本期采用 `RBAC + Menu + Action + DataScope` 四层模型：

1. `Tenant`：租户边界，决定数据隔离和功能套餐。
2. `ContextScope`：运行时上下文边界，分为 `platform` 与 `tenant` 两种模式。
2. `Menu`：页面入口与导航节点。
3. `Action`：页面中的按钮/操作点，如新增、编辑、审批、导出、转单。
4. `Role`：角色集合，对菜单、功能点、数据范围进行授权。
5. `User`：人员，通过角色获得权限。
6. `DataScope`：数据可见范围，如“全部数据 / 本部门 / 本人 / 指定仓库 / 指定客户”。

### 6.2 授权规则

1. 用户可以拥有多个角色，权限取并集。
2. 菜单不可见时，其下功能点默认不可用。
3. 功能点可独立于菜单进行禁用，用于同页不同操作权限控制。
4. 数据范围在角色层定义，在用户层可附加限定项。
5. 系统预置角色不可删除，但可按租户复制为自定义角色。
6. `platform_super_admin` 不直接拥有租户业务数据写权限，必须先切换到目标租户上下文后，才以“平台代管”身份执行租户级系统管理操作。

### 6.3 登录模型

本期统一采用双入口登录模型：

1. `tenant` 登录
   - 适用对象：租户管理员、老板、业务人员
   - 必须输入 `tenantCode + username + password`
   - 登录后直接进入当前租户上下文
2. `platform` 登录
   - 适用对象：`platform_super_admin`
   - 仅输入 `username + password`
   - 登录后先进入平台控制台，无默认租户业务上下文
   - 必须显式选择目标租户，系统生成新的租户上下文快照并记录审计日志

### 6.3 数据范围模型

建议支持以下标准范围：

- `all`：全部数据
- `department`：本部门数据
- `self`：本人数据
- `warehouse_assigned`：分配仓库
- `customer_assigned`：分配客户
- `supplier_assigned`：分配供应商
- `production_line_assigned`：分配产线/班组
- `custom_tags`：预留扩展标签范围

## 7. 租户边界与隔离原则

1. 菜单模板分为“系统模板”和“租户实例”。
2. 系统模板由 `platform_super_admin` 维护，租户只能继承或局部禁用，不可直接改写系统模板。
3. 租户自定义角色、人员、角色授权、人员角色关系必须严格按 `tenant_id` 隔离。
4. `platform_super_admin` 可跨租户查看租户列表，但不可在平台态直接操作租户业务数据。
5. `platform_super_admin` 只有在显式切换到目标租户上下文后，才可进入该租户的系统管理页面；切换、退出、敏感变更都必须记录审计日志。
6. 租户管理员仅能管理本租户的角色、人员、菜单可用性和功能开关。

## 8. 与现有固定角色体系的兼容与迁移

### 8.1 当前固定角色

当前代码中已经固化的角色：

- admin
- boss
- purchaser
- warehouse
- supervisor
- worker
- qc
- sales

### 8.2 兼容原则

1. 第一阶段保留现有 `UserRole` 枚举，作为系统预置角色编码，并新增平台级预置角色 `platform_super_admin`。
2. 所有老用户登录后，若仅存在旧 `roles[]` 信息，系统自动映射到同编码的预置角色。
3. Sidebar 静态角色显隐先保留，同时新增“菜单编码”配置，前端逐步从“角色判断”迁移到“权限点判断”。
4. 后端接口在迁移期同时支持：
   - 旧逻辑：`requireRoles([boss, sales])`
   - 新逻辑：`requirePermissions(['sales.order.create'])`
5. 当某业务模块完成迁移后，旧角色硬编码逐步下线。
6. 现有租户登录链路保持兼容，平台登录和租户上下文切换作为新增链路接入，不影响普通账号。

### 8.3 迁移策略

分三步：

1. 建主数据：先上菜单/功能点/角色/用户角色关系表，并生成预置数据。
2. 做兼容层：登录态返回 `roles + permissions`，前端新增权限判断 hook。
3. 分模块迁移：优先迁移 Sidebar、主数据页、审批页、导出按钮等关键权限点。

## 9. 功能需求

### 9.1 租户配置

支持：

- 租户新建、编辑、启用、停用
- 套餐/功能开关配置
- 默认管理员设置
- 默认角色模板选择
- 租户到期、备注、状态管理
- 平台超级管理员从租户列表显式切换进入目标租户上下文
- 平台超级管理员退出目标租户上下文并回到平台控制台

### 9.2 菜单与功能模块配置

支持：

- 菜单树维护
- 菜单排序、层级、路由、图标、所属分组配置
- 页面功能点维护
- 是否系统预置、是否租户可见、是否默认启用

### 9.3 角色配置

支持：

- 角色新建、复制、编辑、禁用、删除
- 角色类型：系统预置 / 租户自定义
- 角色说明、优先级、数据范围模板

### 9.4 人员配置

支持：

- 人员新建、编辑、启停用、重置密码
- 绑定租户、部门、岗位、手机号、邮箱
- 登录账号与状态管理

### 9.5 角色菜单及功能配置

支持：

- 角色绑定菜单树
- 角色绑定功能点
- 角色设置数据范围
- 可视化勾选“查看/新增/编辑/删除/审批/导出/打印/转单”等功能点

### 9.6 人员角色分配

支持：

- 单人分配多角色
- 批量导入角色分配
- 角色生效时间、失效时间
- 主角色标记

## 10. 非功能要求

1. 后台权限页必须支持大表格筛选、批量操作、审计日志入口。
2. 所有授权变更必须记录操作日志。
3. 授权结果在重新登录后稳定生效；关键功能支持刷新权限缓存。
4. 页面打开、树形加载、角色授权提交应满足常规后台管理性能要求。
5. 平台态与租户态必须在 UI 上明确区分，避免误把平台操作当作租户操作。
6. 平台超级管理员的登录、租户切换、退出上下文、敏感写操作必须完整审计。

## 11. 验收标准

### 11.1 功能验收

1. 可在平台侧创建新租户并分配默认管理员。
2. 可在租户侧创建自定义角色并分配菜单与功能点。
3. 可在人员页为员工分配多个角色。
4. 被取消授权的菜单在导航中不可见。
5. 被取消授权的功能点在页面按钮层不可见，接口层不可执行。
6. 同一账号切换角色组合后，数据范围按最宽并集计算并可追溯。
7. `platform_super_admin` 可无 `tenantCode` 登录平台控制台，但不能在未切换租户上下文时直接操作租户业务数据。
8. 普通 `tenant_admin` 与业务角色登录时仍必须填写 `tenantCode`，且不能切换到其他租户。

### 11.2 回归验收

1. 现有固定角色登录后，原有菜单访问不出现明显回退。
2. 原有 boss、purchaser、warehouse、supervisor、worker、qc、sales 业务路径保持可用。
3. 多租户用户不会看到其他租户的角色、人员与授权记录。
4. 平台超级管理员切换租户上下文后，可见范围严格等于目标租户；退出后不再保留该租户业务上下文。

## 12. 发布约束

1. 权限模块上线前必须先完成预置菜单与预置功能点初始化脚本。
2. 上线必须包含审计日志与回滚方案。
3. 至少完成 Sidebar 与关键敏感按钮的权限点接入，否则只算“主数据上线”，不算“权限生效上线”。
4. 若引入 `platform_super_admin`，必须同时交付平台态登录入口、租户切换入口与切换审计，禁止只上“跨租户超级角色”而不上审计闭环。
